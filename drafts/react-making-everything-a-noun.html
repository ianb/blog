Ian Bicking: a blog

Thursday, February 26th, 2015

React, and making everything into a noun

On a recent project I decided to look around for some more tools to try to keep my client-side/Javascript/HTML UI code in order.

Previously I had stuck with using jQuery and manual manipulation of the DOM. One advantage is that I never got bogged down in framework confusion, and it worked okay, but it was really hard keeping track of the UI code, and some things were really hard or resulted in really tangly code.

One of the things I worry about most with UI technology is how it biases the interface design. I want the UI to ultimately be the best UI it can be, to represent user’s idea of how things should work, to express information in the most understandable way. Then there’s the underlying application, which represents the functionality of the application in a way that is stable and flexible and debuggable. In my experience the two sides are often quite far apart (quite the opposite of Naked Objects).

But when some kinds of UI are easy to implement, and others are hard, it’s easy to start to prefer the easy UIs not because they are easy, but because you can actually write them and they aren’t buggy and you can adjust them more easily. UI code that is easy is the one you can tweak and the one you can grow — so the easy approach becomes the better approach, even if there is more potential in the hard approach.

Writing fully verb-oriented code

When you are using jQuery (or any straight DOM manipulation) you are in a very verb-oriented world. You are always doing things to the DOM. When your UI represents actions this works great: the actions map exactly to some DOM changes. When you application is representing state, and that state updates, then it’s not great. You have to somehow convert state into actions: figure out what is now missing, and delete it from the DOM, figure out how to update derived representations, etc.

I managed to actually keep some of that under control by using generated class names as a kind of data binding. An object would have an ID and attributes, like say if something was derived from an object Person, with ID 123, and the avatar attribute on that person, I’d look for all elements that matched .person-123-avatar. And these Person domain objects wouldn’t really be deleted, they’d just be hidden, and so when I updated the DOM I could easily detect what I had to remove from the DOM. It became like a very ad hoc and limited data binding system.

This worked well for a pattern where my UI was primarily driven by events from the application. Every time something was added or removed or changed there would be another event. Initial state is handled just by emitting all a bunch of add events. Updates remain tricky.

Templating

To make things a little less action-oriented everyone’s first stop is likely to be templates.

Templates are a bit tricky, because you typically have to figure out what to rerender in response to changes. You’ll also have brand-new DOM elements on every rerender, so anything dynamic you’ve attached to those elements has to be reapplied.

Figuring out what to rerender can be tricky. You have to have some sense of what depends on what. You will probably break your templates down into sub-templates, so you can rerender just bits. It’s not like server-side templates where you can have one large template that covers all of the page-specific UI.

To reapply functionality to your rerendered templates you might associate that functionality declaratively so it happens automatically and consistently, paired with something that automatically reapplies the functionality when templates are rendered. You could also rely on event delegation to apply all your handlers to the document, and then dispatch that handling based on event.target.

I have found myself unable to love templating. At best it feels like merely a better hack. I don’t like how I have to break things into chunks and carefully track rerendering. It feels like the manual DOM approach, only the granularity is somewhat larger and you get to use something that looks more like HTML, like <div>{message}</div> instead of $('<div>').text(message)

I also like mushy programming, separation of concerns doesn’t itself excite me. In a specific context, sure, but I’m always skeptical of where we place the separation, and I’m happier throwing the concepts into one big bucket and splitting it out when I’ve reestablished confidence about the right division. I want to avoid too many preconceptions about how a program should be divided. Templates feel like an expedient separation, but not necessarily the right separation.

MVC Frameworks

I can’t comment much on the MVC frameworks because I can’t get myself to use one. Every so often I try and fail.

A general tension I’ve started to feel is verb-oriented UI (all actions) vs. noun-oriented UI (all rendered state), and the complications from choosing either or both. With that perspective I understand more of the decisions Javascript MVC frameworks have made.

I still don’t like those frameworks. The decisions feel like an motley group of approaches, attacking the problem from all directions, even within a single framework. And even if they get make great decisions, it’s still a framework, and I just don’t like frameworks.

An aside on frameworks

What defines a framework, to me, is that in some core way the framework code invokes your code. The framework doesn’t just do things, you don’t just invoke its functions, but rather the framework controls things.

The most innocent example of this is probably the URL router (client-side it might be reading fragments):

var router = Router();
router.add("user/{id}", showUser);
router.add("settings", showSettings);

Here the router calls showUser or showSettings, etc. I don’t hate this, but even here I might happily replace this with a switch like:

var router = Router();
// Now the first argument is a 'name':
router.add("user", "user/{id}");
router.add("settings", "settings");
router.on("hashchange", function (match) {
  if (match.name == "user") {
    showUser();
  } else if (match.name == "settings") {
    showSettings();
  }
});

I know, I’m hopelessly tedious and verbose. But now you know something about my coding aesthetic and can judge as you will.

Of course frameworks are far more complicated than this. There is deep logic going on in the framework before it calls your code. I’m not comfortable with deep logic I don’t personally understand. I’ve written my own frameworks of course, and I understood those, and when I was writing them I thought frameworks were pretty great. But I’m older and wiser or maybe just more conservative, and I’m not excited about frameworks anymore. I know why they exist, but as technologists we must imagine what could be, not only what is, and every framework to me feels like it’s the perhaps-understandable stopgap because we haven’t factored the problem space correctly yet.

This is a bit of a fantasy though — if you don’t use a framework, your application will become a framework. It’s a kind of rephrasing of Greenspun’s Law. So do you want to make a framework or use a framework? The framework you make will be smaller, the person who writes it will understand it, and you’ll be a little more free to debug things at any level (though you will also have your own bugs at every level). I personally prefer to write my own app-specific framework, perhaps because I’m older and wiser and more brash.

But writing your own application, with an ad hoc informally-specified framework embedded in the application, works much better if the framework is small, and you’ve de-frameworked as many parts as you can. Which is why I’ve been pretty excited about React. But I digress…

React: banishing verbs!

So I’ve started thinking about the tension in these designs to be a contrast between action-oriented designs and rendered-state designs. Or verb vs. nouns. Some view it as imperative vs. functional, which steeps the choice in a rich history, but I’m not sure I want to join that old battle.

Since I don’t like MVC frameworks, and I wanted something better than the imperative DOM manipulation approach, and templates have some problems, I became interested in React.

Honestly it was a hunch — I had a hard time figuring out what React actually was. When I first encountered it there was a lot in its description about creating reusable components. But who cares about reuse when you aren’t sure yet that you even want to use something? And what are components anyway? The messaging is a bit more clear now.

I actually understood React when I decided it was just a templating library that addresses the problems I have with using templates. It addresses those problems by taking a much different approach to the separation of concerns than most templating libraries.

The pattern React uses is:

  1. You figure out all the data your UI needs
  2. You give the data to a “Component”, which is more or less like a template
  3. The Component renders the data to a “virtual DOM”
  4. React takes that virtual DOM and makes it so in the real DOM (but making a minimum number of actual updates to make it so)
  5. React basically owns the DOM, or at least the DOM trees it renders, so any interactivity you include has to go through React

That last part is what really makes React feel different, and kind of frustrating, and where something as pleasantly innocuous as a templating language turns into a framework. But unlike other frameworks I think I might be okay with this one.

On the React site they call this approach “data flow”; the data is always flowing from your application and domain objects into the UI. When something changes you always start over from step 1. There’s nothing eclectic about the approach — for instance lists and conditionals aren’t treated specially, sub-Components (aka Partials) aren’t really very different from inlining the equivalent code. It’s a highly consistent approach. (It also seems inefficient, but React goes to great lengths to make the seemingly naive approach fast.)

This is kind of awesome because it replaces all the varied approaches an MVC framework uses. You statically define how some UI renders, and as necessary you rerender pretty much the entire thing.

It also encourages you not to use mutable data structures, or at least to create a kind of immutable view on those structures. (Being single-threaded, everything in Javascript is immutable if your code doesn’t mutate anything, at least for the period of time that code is in control.)

Maybe you can think of it as a kind of transactional UI: everything can change, but nothing is reflected until you tell React that stuff has changed. But you don’t have to be very specific about what changed, on every change you deliver everything up to React.

Reading this post about Om and React framed React well when I started using it — it talks about React from the perspective of a functional programming language (ClojureScript). I’m personally anti-compile-to-Javascript, but I still like hearing from people with alternative perspectives on what the Javascript runtime can be.

But not everything is a noun…?

What I have found challenging about React is the times when I want to create interactions that involve the DOM, but don’t feel like “state”. If I have a button that expands out to show options, the “show” state isn’t application state, it’s UI state.

I have not figured this out. I mean, I can make it work, but I don’t like what I’ve done. Maybe I’ll gain confidence on techniques with experience, but I worry that with experience I’ll simply get good enough at doing things the way React wants me to, and I’ll forget that it’s actually weird. This is the same way functional programmers start to alienate themselves from mainstream programming — they get good enough at it that they forget what they are doing is weird, and maybe start to think “weird” is just subjective. On principle I do not want to take this approach — there’s bad weird, there’s good weird, but it’s no good to go off in a corner and say that the rest of the world is weird and you are the normal one.

But I digress. Because React assembles and manipulates the actual DOM, and attaches any handlers, you have to go through it most of the time. And when you don’t it gets tricky.

If you do it the React way you will find a way to create input data that renders whatever it is you want to render, meaning you turn every verb into a noun. All side effects become an aspect of the state you use to render the UI.

(Notably though you don’t need to use React on everything — I find the main layout of the page to be easier to write outside of React, for instance. But everything in a specific DOM tree that React renders will have to be React-aware at least.)

JSX

React is essentially a DOM-building technique. If you want to render some message, you might do:

var Message = React.createClass({
  render: function () {
    return React.DOM.div(null, this.props.message);
  }
});

That’s more-or-less equivalent to $('<div>').text(message). And it doesn’t look like HTML, and that’s no good. So React includes its own little Javascript dialect where you can do:

var Message = React.createClass({
  render: function () {
    return <div>{this.props.message}</div>;
  }
});

At first I thought I’d stick with the ugly but unmagical first form, but I quickly put aside my code generation biases and used JSX. Conveniently Emacs supports E4X which is not the same as JSX, but is syntactically the same, and makes editing the code more pleasant.

While the code is Javascript, I treat it as a template and keep it in a separate file from me other application code.

I’ve appreciated having the full expressiveness of Javascript. My only real annoyance so far has been that JSX isn’t HTML, and worse it’s for trivial reasons — you have to use className="foo" instead of class="foo" and style={'background-color': 'red'} instead of style="background-color: red".

Thoughts on user interface design

I like that React gives me something that isn’t really a framework, but removes the need for lots of things a framework does. But just as important it positively affects what is easy to do with your UI.

Where this was most noticeable was in the activity stream we implemented. An activity stream seems very simple and easy to represent as a series of template partials. But there are rules that aren’t so simple:

  • Time-sorted seems natural, but you might want to keep some things above the fold regardless of proper sorting.

  • We wanted to reveal some items in only certain circumstances. For instance, when showing browser activity it’s boring when you go to about:blank — but if that’s your active tab then we want to show it anyway.

  • Adjacent items can sometimes be combined.

  • As things get old we might want to abbreviate them.

  • We might want to note some context that isn’t contained in a single data item — for instance, when something else in the stream relates to another item, we might make a note in each about the other.

Using React the code for any of these is not hard. But each change breaks down the one-to-one relationship between application objects and the DOM. You can’t use a template partial. Template partials work great if you have a set-it-and-forget-it approach, where you just pile things onto the stream and never update them. But when pieces interact the other techniques are hard. With React it’s easy.

In my opinion good UI reflects the user’s conceptions above the application’s conceptions. If you do this the UI can easily drift away from the application’s domain objects. And you want to let them drift. If you do then your UI layer has to handle the complexities of this mismatch.
