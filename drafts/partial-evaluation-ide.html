Partial evaluation in the context of an IDE

Partial evaluation is a technique where you evaluate only what you can. It takes a program, and creates another program with some of the work done ahead of time. Typically this replaces expressions with static values, with the result of that expression. For instance 1 + 1 could be replaced with 2.

Here’s a possible example:

function createElement(tagName, attributes, children) {
  let el = document.createElement(tagName);
  for (let [name, value] of Object.entries(attributes)) { // Object.entries() iterates over [key, value]
    el.setAttribute(name, value);
  }
  for (let child of children) {
    if (typeof child === "string") {
      el.appendChild(document.createTextNode(child));
    } else {
      el.appendChild(child);
    }
  }
  return el;
}

function userWidget(username) {
  return createElement("span", {"class": "user"}, [username]);
}

Now let’s partially evaluate userWidget():

function userWidget(username) {
  let tagName = "span";
  let attributes = {"class": "user"};
  let children = [username];
  let el = document.createElement(tagName);
  for (let [name, value] of Object.entries(attributes)) {
    el.setAttribute(name, value);
  }
  for (let child of children) {
    if (typeof child === "string") {
      el.appendChild(document.createTextNode(child));
    } else {
      el.appendChild(child);
    }
  }
  return el;
}

This is just kind of a simple inlining of the createElement function, but this is where we suddenly have a bunch of static values, and can reduce. First we’ll substitute tagName, attributes, and children with their original expressions:

function userWidget(username) {
  let el = document.createElement("span");
  for (let [name, value] of Object.entries({"class": "user"})) {
    el.setAttribute(name, value);
  }
  for (let child of [username]) {
    if (typeof child === "string") {
      el.appendChild(document.createTextNode(child));
    } else {
      el.appendChild(child);
    }
  }
  return el;
}

Now we can unroll the loops, since we know exactly what we are looping over:

function userWidget(username) {
  let el = document.createElement("span");
  let [name, value] = ["class", "user"];
  el.setAttribute(name, value);
  let child = username;
  if (typeof child === "string") {
    el.appendChild(document.createTextNode(child));
  } else {
    el.appendChild(child);
  }
  return el;
}

In this code we don’t know if username is a string or not, so we can’t actually evaluate if (typeof child === "string"), though with a little type annotation we’d be able to reduce this to:

function userWidget(username : string) {
  let el = document.createElement("span");
  el.setAttribute("class", "user");
  el.appendChild(document.createTextNode(username));
  return el;
}

There’s a potential performance win here, but that’s not what I’m interested in. It’s an interesting view of the original code: it tells us what the code is actually going to do. It lets us peek into other functions, and do so in a way that’s specific to what userWidget() is doing.

As programmers we frequently do this mental execution and simplification of code: winding through execution, using inferred knowledge of the state to eliminate possibilities, and identifying what code really matters in a certain context. Sometimes we do this manually by doing step-through debugging.

So let’s imagine a “static explorer”: static in that you don’t have to execute anything.

Instead of “step in” you have “inline” and “evaluate”. Inlining expands your code, bringing in code external to your immediate environment. Evaluate simplifies the code.

Idly checking out the Wolfram Language. I come upon this page, which is not particularly notable: https://reference.wolfram.com/language/ref/Sow.html

What is notable is this example:

  In: Reap[Sow[a]; b; Sow[c]; Sow[d]; e]
  Out: {e, {{a, c, d}}}

It’s a funny language feature that I haven’t imagined before, but that’s not really the point. The point is that this is a runnable example – that’s actually what is output. Typically when you run something like that you get NameError: a is not defined. And “a” here is really a variable, of sorts – it’s not a string or symbol (at least not a symbol in the sense that we know them in programming).

Given this, snippets of code are just as executable as entire programs. Every expression is like a function with the free variables as its parameters, and a sequence of expressions is a bit like function composition.

This is all natural from the perspective of mathematic notation. In a more traditional programming environment I think it’s reminiscent of partial evaluation: https://en.wikipedia.org/wiki/Partial_evaluation – where you analyze a program and execute expressions opportunistically. It’s really almost the same as partial evaluation, but the Wolfram Language knows a lot more about how you can execute different combinations of expressions than a typical language. A typical language does not really “believe” that (a+b) and (b+a) are equivalent. It doesn’t know how to relate different operations. Nor do normal languages have a concept of simplification, so they can’t speculatively try other arrangements (where none in isolation is clearly better or simpler than another) to see if simplifications are possible.
