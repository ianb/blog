Thoughts On Richer Code

The Future Of Programming by Bret Victor came up again and got me to thinking (again).

In this case the discussion got me to thinking about how we represent code. Why do we stick with ASCII?

One of the biggest reasons: we don’t have anything better. And it shouldn’t be surprising that it turns out that the visual communication we’ve developed over literally thousands of years (writing!) is a good way to communicate to computers.

While visual block-based programming has improved in the last few years, I don’t think they are the basis for a new paradigm. It’s more like good scaffolding.

Maria – a kind of notebook for Clojure code – got me to thinking. I myself have been playing with visual representations of objects. I’ve also been interested (from afar) in the Wolfram language (aka Mathematica).

Anyway, these are the sources of some thoughts…

  1. Take the exploratory notebook approach as our interaction paradigm
  2. Programming doesn’t stop with code; include execution
  3. Programs don’t stop at output; include understanding
  4. Programs must explain themselves beyond code; output must also explain itself

Each of these are really post-hoc justifications:

  1. Let’s not worry about a programming language being “complete”: it doesn’t have to be written in itself, it doesn’t need to be applicable to every problem
  2. Output is the interesting thing, and we treat it as a side-effect, so let’s not do that
  3. The results of programs shouldn’t wash away the program, programs shouldn’t just “stop”
  4. Debugging and testing shouldn’t be separate, those abilities should naturally emerge

So here’s a language I imagine:

  1. The language is based on expressions and expansions of those expressions.
  2. The language is functional, because per the previous point it must be functional; only in a functional language can you expand expressions arbitrarily
  3. The “output” of the language is a fully expanded and resolved expression.
  4. Each expression knows how it was created
  5. Objects and expressions are the same thing. Every object has to have an accurate representation as an expression.
  6. Objects can have non-expression representations as well. The DOM is a natural fit.
  7. Other representations should be inspectable, and that inspection be traceable to the entirety of the object, and how it was constructed
  8. Not everything is amenable to functional programming. For instance, sometimes you have to fetch something. Acquiring information like this should be “built in” in some fashion. Typically you acquire and save the information, for replayability. For instance if you fetch a web page, you will naturally generate an object “page http://foo.com generated at time T” that will be reused for later executions (barring explicit interference)
  9. In other cases, different kinds of input are expected. Like if you are asking a question. In those cases, the input should know what kind of things it may produce. Then we can expand several examples, to show several “executions”.
  10. Every expansion of an expression should be a deterministic and repeatable product of its inputs.
  11. There is text input, probably more or less. But it is rewritable, it is not sacrosanct. You can reenvision it as other representations. You could inline any kind of object, in some theoretical editor.
  12. Maybe it has free variables, instead of defined variables. Placeholders can exist anywhere. You can write expressions inside-out, outside-in. Packaging things up as functions isn’t necessary.
  13. You can have multiple versions of any variable, including functions. This allows all kinds of side-by-side comparisons. This is similar to an undetermined input, and an execution that depends on the input, but instead we’re substituting bits of code.
  14. Maybe the “could be one of many things” would be another kind of container type. Imagine, for instance, X = one_of(1, 2, 3): X in this case is an integer, but everyplace you use X you get three expansions. Given several such variables you may end up with an infeasible combinatorial expansion, but we can fall back on sampling.
  15. = means is, not set. It’s a more mathematical notion.
  16. Clearly expressions should be expressions, i.e., it should be easy to express expressions.
  17. Functions might be polymorphic, but their requirements might be much more than “usual”. E.g., + might require commutivity. They aren’t just symbols. This is because the expression can be rewritten in more complicated ways than just expansion.
  18. Clearly functions can “see” into their arguments, i.e., they are more than just values. So a * (c + b) could be turned into a * c + a * b.
  19. Maybe the ability to see into expressions means packing and unpacking arguments can be done aggressively, so keyword arguments could be dictionary arguments?

I’m initially drawn to Lispy syntaxes. Given the constraints the S-Expression is appealing. But I dislike positional arguments, they aren’t very humane for most purposes, and scale poorly. Heaping keywords onto the S-Expression is ugly. As a result I’m inclined to look more at a Smalltalk-style language, where parameters are labeled.

Objects (at the level underneath the language itself) probably require a fairly rich and mandatory interface. Objects have to describe themselves in several ways.

Expression simplification is an interesting and useful thing. That it go to the length of Mathematica or Maple seems unnecessary. But expansion on top of expansion is not great. At its simplest this is just partial evaluation.

Can objects be instantiated with unknown values? E.g., circle radius: X (where X is unknown)? Probably. Even better if the rendering has visual patterns for unknowns.

This is not a good language for general computing. There’s no zero-cost abstractions here! But that doesn’t mean it can’t be useful, or that it’s a toy, or educational.
