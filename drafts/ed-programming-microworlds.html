Education, Programming, and Microworlds

There’s a lot of debate about whether everyone in the next generation should learn to code, and if so why (for example). And though I have a personal attachment to the educational value of programming, I also get it: we don’t need to turn kids into little workers, programming isn’t for everyone, and and there’s some weird futurist fetish behind the drive to teach programming.

But programming is also unique: you tell the computer what to do and it does it faithfully. It’s not like a graded essay where you are trying to satisfy someone else. No one else has to evaluate your work, and instead of trying to replicate the expected result you can generate truly new results.

One thing I liked from the book Mindstorms (the freely-available book) was the concept of Microworlds. (Both names have since been turned into product names, but seemingly unrelated to the actual ideas.)

Turtle graphics is one Microworld, specifically based around geometry. Hence its relatively novel use of angles and lengths instead of coordinates.

Each world gives people something to explore. That they do it programmatically is interesting, and allows for different kinds of exploration, but it’s the world they are exploring, not the code.

Mindstorms hints at the idea that there should be many such microworlds. Maybe they all use Logo, or maybe not.

What didn’t happen is all those other Microworlds in Logo. But some have come along: Processing (https://processing.org/) is one example. And I think it’s notable that “Processing”, the graphical approach, is treated separately from any particular language implementation; Processing.js is another valid implementation of the same microworld.

But there could be many more! And I think it’s much more interesting than yet another puzzle-based block programming language to teach coding.

  1. Choose Your Own Adventure approaches to nonlinear fiction, which starts without any programming but easily opens up into all sorts of other complexity. Twinery is a good implementation. The first thing you teach isn’t what a variable is; the first thing you teach is how to tell a story, then how to tell a story that collaborates with the reader, then maybe how to use programming to make the story itself autonomous.

  2. Music generation. FoxDot (itself based on SuperCollider) is somewhat accessible. The first concepts here are signatures and beats and intervals, and later programming comes in with naming and sequencing of operations.

  3. Lots to be done with procedural generation. Something as simple as cartoon face generation is an easy start (for example. Some of the fun is being able to do some things easily, but digging into the underlying code to add new parameters or change the way things work. But starting at the beginning (like with variables) hides the fun and interesting part behind graphics coding.

  4. Simulators can teach a domain (city design, ecology, electoral politics, etc), but like procedural generation adding some programming can make the exploration more exciting. NetLogo explores this area.

  5. Eliza, in her original implementation, is like a very simple programming language for interaction. The Rogerian psychotherapy stance that makes Eliza plausible is reminiscent of how a teacher might interact with a student, and I can imagine a child implementing a “teacher” chatbot.

  6. Quiz generation. Writing a program that creates a math worksheet seems like automating educational tedium, but I think a lot of kids would find it both interesting and satisfying.

I’ve put in examples of all these things, so in a sense this all exists. But they usually lack appropriate introductions and humane environments designed to work for children. Often they put the programming environment first, instead of situating programming in an environment that has its own educational value.
