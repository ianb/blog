A Personal Opinion On Mozilla’s Product Direction

Mozilla is at a turning point. We’ve just laid off a bunch of people. This feels sad and disappointing. Firefox’s marketshare is all disappointment, and the many real technical improvements we’ve made to Firefox can’t seem to turn it around. And yet Mozilla seems as necessary as ever, if not more.

The sense of loss in the layoffs could turn into a sense of doom, or a need to get out, or resentment, or questioning of means. I would rather think of it in terms of resolve: this is a signal that we can’t just keep fighting the same battles. On a personal level we’re not suffering, Mozilla treats its employees well, and I am confident that the people leaving – and myself when that time comes – will do well. It will be stressful, and leaving against your will is powerfully disempowering. It seems fair that those of us still here also take on a sense of (productive) stress.

The question we have is: what should Mozilla do? As described in the announcement of layoffs, we are making sacrifices to leave room for new opportunities. But any opportunities available are only the ones we find for ourselves.

This is where I will offer my opinion on what product directions Mozilla should take:

User Agents as both tactic and strategy

A user agent is a program that belongs to the user and acts on behalf of the user. Specifically it interacts with third-party services and software.

User agents are fairly uncommon. Browsers of course. Email clients. There used to be instant messaging clients, but these are mostly gone. Mobile OS’s fill many roles of a user agent.

An independent user agent is particularly important. You might consider the Facebook app to be a user agent that allows the user to interact with the Facebook service, but there’s no real boundaries: the client, the backend, the site, the social network, the business that funds it, are all one big thing. The strategy tax rate is very high in a system like Facebook. The user cannot separately choose their agent and their service, which is antithetical to an “agent”.

Because a user agent acts on behalf of the user, it should aspire to interpret the person’s intentions.

Independent user agents are naturally subversive. For instance, ad blockers subvert the monetization of the web, but not in the service of any alternative: they add directed instability (as opposed to directionless instability), and we don’t know how the system will ultimately react. Right now technology (and maybe capitalism generally) is filled with tensions, but the tensions are externalized away from the commercial institutions and onto the public. An independent user agent does not accept that compromise.

Adversarial Interoperability

Adversarial interoperability describes making things interoperable despite the wishes of hopeful monopolists.

As described, some classic examples of adversarial interoperability are the various applications that learned to read Word documents, Samba reimplementing the SMB protocol, and web crawlers.

Sometimes this stuff was dangerous and unwanted. There was a time when it was common for applications to ask for your login credentials to Gmail so they could comb through your email, purportedly only to collect contacts, but who really knows? The answer to this has been APIs, a kind of cooperative interoperability, but whose bounds are limited by both parties’ incentives and ability to coexist in peace.

I believe a user agent can be an important component in both reforming the dangerous old ways and unwinding the silos of today. I wouldn’t want to give some new social network access to my entire email interface and identity, but if I can trust a responsible user agent I believe it’s possible to create a user agent that can mediate that kind of data transfer safely and responsibly confirm intent. By placing a person at the center, and by creating a user agent that aligns itself only with that person, we can create something with unique technical capabilities – freed from the complicated business relationships of rivals – and with the moral standing to make humane choices that may not align with corporate goals.

What might the next user agents look like?

Importantly the next set of user agents need not and should not wait for the world to catch up. From day one a new user agent should work with the world we have right now.

Creating something that works with an uncaring and possibly hostile world is hard. We’ve built browsers on specifications and negotiations, the careful compromise between users and site owners and implementors. I don’t think we’ll have that luxury for this set of user agents, even if the relationship is ultimately the right thing for the entire market.

Some examples that come to mind:

  • A multi-protocol messaging system
  • A promotional mail aggregator/forwarding system
  • Account management: list accounts you have, offer to delete fallow accounts, etc. (Features Lockwise may grow.)
  • Tools to allow you to navigate topics regardless of linking; e.g., if I see a questionable article, can I find trusted articles on the same subject?
  • Ways to navigate multiple video streaming services
  • Tools to move data between personalized or authenticated services
  • Collective aggregation and analysis of personalized advertising (sometimes users want to make data public that a service provider would rather keep private)
  • Firewalls and VPNs and other tools that can mediate at the network level
  • Tools to interpret difficult or opaque documents, such as terms of service agreements.
  • Tools to navigate customer service relationships. Put a bot at both ends of the chat!
  • Tools to publish, aggregate, and analyze bills. Start with cable and internet service.
  • Apply third-party indexes, reviews, and commentary to other services.

Some of these are examples of re-establishing a latent client/service relationship. Others try to pull out structured data where it might otherwise be hard to find and use. Some involve creating collective knowledge, and/or encoding knowledge into the tool.

I think encoding knowledge in our tools is interesting, though relatively untested. Armchair punditry often declares: people just need to start acting smarter! Do your own research! But if positive change requires everyone to suddenly become smarter, more knowledgeable, wise in the ways of every corner of our vast economy and ecology, then that change cannot happen: if a recipe for change is a recipe where everyone becomes something they are not, then it is no recipe. But a trusted agent can be knowledgeable on your behalf, change can happen through reputation and aspiration applied voluntarily to individual’s actions.

Agents that know things on your behalf might sound patronizing and dictatorial, but it’s not far from how things work already. Do we teach everyone how to read domains carefully so they know the difference between paypal.com and paypal.com.service.biz? Or do we put up a warning when someone tries to open the second link?

Keep the web at the center of Mozilla

I believe Mozilla should still go long on the web.

I’ve long felt a pervasive futurist-pessimism about the web, and yet the web still dominates. The web is the source of knowledge, the one common denominator of interface, and by percentages other things grow faster, but in mass the web still grows fastest of all. Sure the web struggles in places, but it’s the water we all swim in.

The web is also so damn malleable. If we’re willing to get our hands dirty then just about everything we need to mediate, separate, aggregate, everything we need to disrupt or enhance for users is in the web somewhere. And Mozilla has the skills.

My own work

A while back I joined the Voice Products team in Emerging Technologies. I’ve been meaning to write about it, but now I’ll have to write about it here.

The team’s definition (voice) made me a bit reluctant at first. I don’t like interface-driven divides. I wasn’t part of the clicky products team previously, nor would I call mobile a touchy browser, and “voice” feels like it should be a secondary attribute. But teams need a name, and the team is open minded, so Voice is also a fine name.

Due to technological advancements voice interfaces are feasible where previously they were not, so there is some reason to think opportunities are newly available. But potential opportunities are only as good as the actual opportunities you find.

What excites me particularly about voice is how directly it engages with intention. Every visual and click interface is a kind of multiple-choice-question. There are still a lot of choices! But you can’t click on something that’s not on the screen. And it’s not just the finite set of options, but how those options are composed: user intention is performed through a set of interactions. With your voice you can say anything. You can talk about things that don’t exist yet, things in the past that have paged away, you can combine thoughts and actions, nouns and verbs in all kinds of ways.

Of course you can’t actually do many of those things right now. But what a fun thing to try to do! There’s open space to explore. And all of it is engaged tightly with intention: unbounded by multiple choice interfaces we can imagine a person saying what they really want to do instead of manually executing on a recipe of actions to achieve an intention.

And because you can talk about things you can’t see, you can express intents that aren’t bound by services. Each tab in a browser is its own world, and cross-tab/cross-site/cross-service interactions require the clipboard and tab switching and a lot of personal mental accounting.

We’re starting out with a voice assistant in the browser. The ceiling on this particular packaging of voice functionality is probably low; there are limited use cases for talking to your computer. But I am hoping we can retain a sense of imagination to see how this is the scaffolding for much more.

Right now we cannot actually do most of the things you’d want to tell your browser to do. We’re adding stuff, but it takes time, it’s a very small team, and we’re very much trying to understand what exactly it is we (and users) are trying to accomplish. By attaching our tool to a high-functioning and familiar web browser we will never abandon a user, because they can always just go back to normal user. As we get get better I hope the alternate input methods become less and less necessary. Speech output could in turn makes the visual output less necessary.

If you don’t need to see the browser, if we can mediate and abbreviate the necessary output, if we can understand the commands you give, I can see how these same approaches can be applied to mobile or a voice assistant or a TV or whatever else.

If the tool can confidently execute even more (or at least know its own limits) then you can take those same actions that a person speaks and start to do them on behalf of the user even when they aren’t present, creating an autonomous agent.

If all this is built on the web, and even built on browsers, at each stage you can fail back to another familiar interface, or provide auditable interactions like a set of screenshots that show how the voice assistant tried to perform an interaction. At no point does it have to be perfect, just better and better.

There’s so many unknowns here, and hard and very much unsolved problems, that we can’t know if it’s possible. This is one possible user agent among many, and Mozilla should try a bunch. But I’m optimistic. And pessimistic: actually accomplishing this will require some faith, some stubbornness, some humility, some ingenuity, and definitely a lot of community. That’s a lot of things that have to come together in order to achieve success. For this experiment, or any other product experiment Mozilla will try, we need to figure out how to be a bit more brave.
