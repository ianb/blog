A Program Bigger Than Its Text

A couple new articles on live programming:

  • Working toward Bret Victor’s “Learnable Programming”
  • A Live Programming Experience

Bret Victor’s Learnable Programming emphasized these visual environments, which feel like a red herring to me. In the first article about Alive they say: Live programming research should focus on making “boring” code easier to read and understand – I think that’s a good way to think about it, as we want to spend less time on the boring problems, we want to push those problems aside and get to the meat of the program, the interesting and hard and vague problems we are exploring. These aren’t visually interesting problems.

A digression: in a The Future Of Programming Bret compares binary programming (directly coding machine instructions), to assembly programming, to “high” level programming (such as Fortran). He describes Von Neumann criticizing a student for writing an assembler, as it would waste valuable machine time doing the assembly, which is the job of these clerical programmers. You can view the change to higher level programming as a change in economics: computer time became cheaper, and so we let the computer do more work. We could also say the computer is less likely to get things wrong. But there was a time, not impossibly long ago, when assembly was seen as awkward, but still better if you are willing to invest the time. More efficient and faster, even if challenging to write and get right.

These days we know that you probably won’t make something faster and more efficient by getting down to assembly. The machine can look at the entirety of the program, and apply optimizations that you could not have made in isolation, or which might be too tedious to apply. Or optimizations that are fragile – but the computer will faithfully and without complaining remove the optimizations when you recompile with changes that invalidate that optimization (it’s much harder to get a human to do that, especially without complaining). Higher level languages let the computer know more about what your intention is, and let the computer take over some parts.

We’ve been making yet higher level languages, but it’s the same approach. I think the intention of “live programming” (the term implies a specific approach, I infer an intention behind that) is about letting the computer take over even more, qualitatively more, different than just ever-higher-level languages.

The essential thing is that the computer take over exploring your program. A visual representation shows how your drawing program runs. All programs run, even the boring ones! The live programming examples typically are very deterministic, and we see expectations that the computer can exhaust its exploration of the program. But I imagine something more incremental, less thorough, and more change-oriented. Let the computer simply explore, and point things out.

Some challenges:

  1. How does the computer explore a program? The problem is similar to fuzzing, the approach where you test a program with random inputs, but we need to give the computer more hints in order for it to successfully find interesting cases.

  2. We need repeatable and reliable runs of programs. A “program” can be a system, several things working together (e.g., a web server, database, and running web page), but we need to be able to reset and predictably repeat runs of that programming system.

  3. We need to be able to represent a run of the program to the programmer. If the computer finds something, how can we understand what it found? I suspect this will take some programmer work, specific work to show how to represent some of the domain objects in compact and useful ways. Like adding __repr__ methods. But still even with programmer help we need to assemble this all into a large time-spanning representation.

  4. The computer needs to know what is interesting. Sometimes it might be “runs 2018 and 19402 changed or were invalidated by your edit”. But I think forms of self-testing will be important. Such as the assert statement or even simple warning statements. “Probes” are another approach. What changes this bit of UI? Maybe add global watch points: I’ve heard this peculiar behavior happens, show me all the cases where it does.

  5. The computer should know why things happen. So you’ve found an interesting case, how did it come to be? Fuzzers work hard at this to minimize their input. What a fuzzer finds after ten thousand steps may only require a hundred steps to reproduce. This is one form of answering why. Another is embedded in that representation of how the program ran. Can we create a browser of program state? Did we hit record before we ran the program?

Each of these problems is somewhat distinct, and can be approached independently. What I think brings it together is looking at the execution of the program, or of an entire system, and not focus only on its source code. A way to look at all the emergent properties of a program; and we want emergent properties, that’s why we right programs and don’t just directly produce output.
