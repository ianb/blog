Experimenting on Contextual Recommendations

A proposed development approach for contextual recommendation

I want to distinguish contextual recommendation from New Tab recommendations: a contextual recommendation asks: how can Firefox help you here, on the page you are on. Some things that make this different:

  1. The user does not necessarily want to do anything new, they might be happy doing exactly what they are doing.
  2. The user does not necessarily know what they want to do next.
  3. Imagine the moment before you focus on the URL bar and enter something, the moment before you create a new tab to do a new task. There is a yet-undetermined readiness, a moment to positively engage the user in a different pattern.
  4. Imagine the moment when you know there is something you should do, somewhere you should go, and then instead you go to facebook.com or some other distraction.

Really we’re on new ground here: major internet properties are built as a distraction engine, and if we want to do something different than be a Better Distraction Enabler we have to figure out what it is we’re even trying to do. (There’s a product question here too: do we imagine ourselves the Fox News of browsers, creating material that pulls you through the commercial to the next item; or do we imagine ourselves the Public Radio of browsers, where the material is intended to make you feel good about your experience after you’ve completed the experience.)

Start with history

The best corpus we have for these activities already exists in browser history. My personal Places history has about 100,000 URLs. This list is of course biased towards my own activity, but it’s fairly easy to pluralize that bias with just a handful of contributing developers. If we build off some other set we’ve introduced a bias that isn’t rooted in even one person. Hand-wavy use cases are highly biased, usually towards “interesting” activities. 1/3 of the URLs in my personal history are to mail.google.com (not something I would have expected).

Access to personal history is also the weapon we will bring into battle. We want to have access to aggregate data, but we don’t, we don’t know when we will have that access, we don’t know why we should have that access, and we don’t know the value we will trade for it). But history is ready for us!

History also points us at personally-valuable content that has no public web presence. This content is the content that only a user agent has access to. This is our unfair advantage. We can scrape, process, re-present this information when no one else can.

Big and annoying UI

I think we need to claim some space for this contextual recommendation. This is not to assert that the “right” product will be big and bold and in-your-face. Rather this is for our own consumption:

  1. As developers we should be constantly aware of what we are creating!
  2. If there is the seed of value, we should be ready to see it
  3. If there isn’t value, we shouldn’t hide that from ourselves, we should be confronted with it
  4. As developers we need extra space to understand why we are getting results, because we have the ability to work backwards given that information

This is based on the assumption that nothing we make now will be right, that our goal right now is learning and experimentation, and building the scaffolding for that learning.

Retroactively apply techniques to history

In my limited experience (a couple days of hacking), it’s difficult to get enough information from what gets picked up from new browsing. Instead we want to retroactively apply our ideas to old history. Some of this information we can pick up from Places (like URLs and some referrer information), but in other cases we want access to the content itself. I’ve started to create a system to pull in that data, from an authenticated personal profile (though the current version of the page, not the historical version). By doing this we can have copies of history that can be re-parsed relatively quickly, allowing us to switch up approaches with little penalty.

If we want to look at behavioral data in addition to content data it would be helpful to factor in whatever we can construct from Places, even if it is of lower fidelity than what we can get from new behavioral monitoring. Places currently supports very “flat” access, but it contains enough information to construct a browsing graph in many cases.
